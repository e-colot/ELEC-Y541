\section{Introduction}

The objective of this report is to showcase the work done in an effort to implement an autoencoder (AE) architecture on an FPGA using high-level synthesis (HLS) tools. The model uses the MNIST dataset, consisting of 28x28 black and white pixels of handwritten numbers, and attempts to compress those images while minimizing the difference between the original image and the reconstructed one.

A trained model, written with PyTorch, was already given and only inference had to be run on the final implementation. The model first had to be translated in C++, which is explained in Sec. \ref{sec:c++_implementation}. The resulting implementation then has been tuned with different synthesis directives, optimizing either the performances in Sec. \ref{sec:layerwise_optimization} or the area in Sec. \ref{sec:ae_optimization}.

\section{C++ Implementation}
\label{sec:c++_implementation}

\subsection{Auto-encoder Model Structure}
The model consists of 4 encryption layers and 4 decryption layers. Each layer consists of a linear part where the input is multiplied by a \textit{weights} matrix, and then added to a \textit{bias} vector. This linear operation is then followed by an activation function, which is a rectified linear unit (ReLU) for all layers except for the output layer.

Each layer can therefore be implemented by calling \lstinline[style=cppstyle]{Layer} (given below) with the right sizes and weights/biasses. It can be mathematically summarized as
\begin{equation}
    y_i = \begin{cases}
        0 \hspace{2.53cm} \text{if} \quad (\mathbf{W_{(i, :)}} \mathbf{u} + b_i) < 0\\
        \mathbf{W_{(i, :)}} \mathbf{u} + b_i \qquad \text{otherwise},
    \end{cases}
\end{equation}
where $\mathbf{u}$ and $\mathbf{y}$ are respectively the input and output vector, $\mathbf{W}$ the weights matrix and $\mathbf{b}$ the bias vector.

\begin{listing}[htbp]
\label{lst:layer_nostream}
\begin{minipage}{\linewidth}
\begin{lstlisting}[style=cppstyle]
void Layer(float layerInput[IN_SIZE],
            float layerOutput[OUT_SIZE],
            float weights[OUT_SIZE][IN_SIZE],
            float bias[OUT_SIZE]) {
    L0:for(int row = 0; row < OUT_SIZE; row++) {
        L1:for(int col = 0; col < IN_SIZE; col++) {
            layerOutput[row] = layerOutput[row] + weights[row][col] * layerInput[col];
        }
        layerOutput[row] = max(0.0f, layerOutput[row] + bias[row]); // ReLU
    }
}
\end{lstlisting}
\end{minipage}
\end{listing}

\subsection{Layer Specialization}
\label{sec:layer_specialization}
A first issue that arose from the basic approach of a single function called for every layer is the dynamic number of iterations in the \textbf{L0} and \textbf{L1} loops, preventing loop pipelining as it would need to have an undetermined number of stages. For this reason, each type of layer\footnote{a \textit{type of layer} designates a layer with a given input and output layer size.} has a specific function with fixed sizes.

\subsection{Stream Model}
\label{sec:stream_model}
For area optimization, FIFOs should be used whenever possible. To allow to have only FIFOs in a layer, each input and output must be read from / written to a single time and in a sequential order. For this reason, \lstinline[style=cppstyle]{Layer} has been written with a buffer for the input as it was the only variable being used not sequentially

\begin{listing}[htbp]
\label{lst:layer_stream}
\begin{minipage}{\linewidth}
\begin{lstlisting}[style=cppstyle]
void Layer(float layerInput[IN_SIZE],
            float layerOutput[OUT_SIZE],
            float weights[OUT_SIZE][IN_SIZE],
            float bias[OUT_SIZE]) {
    float in_buffer[IN_SIZE];
    float tmp;
    L0:for(int row = 0; row < OUT_SIZE; row++) {
        tmp = bias[row];  // Start with bias
        L1:for(int col = 0; col < IN_SIZE; col++) {
            if (row == 0) {
                // Save to buffer if first encounter
                in_buffer[col] = layerInput[col];
            }
            tmp = tmp + weights[row][col] * in_buffer[col];
        }
        layerOutput[row] = max(0.0f, tmp); // ReLU
    }
}
\end{lstlisting}
\end{minipage}
\end{listing}

\subsection{Autoencoder Code}
Apart from the layers, the full code contains a first flattening operation, which converts the loaded image from a 28x28 matrix into a single vector of 784 elements and that removes the input mean and variance. \\
The opposite is applied at the output, reshaping it in a matrix. Those two functions are not shown here as they don't bring anything useful to the report.

\section{Layer-wise Optimization}
\label{sec:layerwise_optimization}
Before optimizing the whole model, a single layer was optimized using hls directives. This gave a more precise understanding of the impact of the used directives.

\subsection{Baseline}
The baseline was taken by disabling pipelining and loop flattening on both \textbf{L0} and \textbf{L1} loops. It used the code introduced in Sec. \ref{sec:layer_specialization}.

\subsection{Performance Oriented}

\subsubsection{Inner Loop Pipelining}
By pipelining the loop \textbf{L1}, each iteration tries to be completed in a single cycle (II = 1 by default) to process the next iteration right afterwards. \\
There is however a feedback in the loop as the output of the layer is updated for each row of the input. The math being run in floating point, the addition takes multiple cycles and the resulting design can only assure an initiation interval (II) of 3 cycles, as shown on Fig. \ref{fig:layer_inner_pipelining_ii}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{pictures/layer_inner_pipelining_ii.png}
    \caption{Schedule of the inner loop when pipelined \textbf{L1}.}
    \label{fig:layer_inner_pipelining_ii}
\end{figure}

An iteration of the whole loop taking 9 cycles, the pipelining should result in a division of the interval by 3. As the outer loop is not modified, the latency should be improved by the same factor and stay similar to the interval.

\subsubsection{Outer Loop Pipelining}

\textbf{L0} can be pipelined in order to reduce the interval. This will unroll the inner loop as a pipelined loop must not contain any other loop. The function will now compute a new row of the output at each cycle, making the interval drop to \textit{OUT\_SIZE}\footnote{equal to 256 here}. \\
Because \textbf{L1} is unrolled, each of its iterations is performed simultaneously. This means \textit{weights} and \textit{layerInput} should have multiple read operations happening at the same cycle. To do so, these arrays are partitioned along the accessed dimension, which is the first one for the input layer and the second one for the weights matrix. \\
The resulting latency can be computed as the number of pipeline stages, to which the number of cycles of a single stage must be added. Fig. \ref{fig:layer_outer_pipelining_ii} gives an insight on how the inner loop has been flattened. As the operation is a multiplication followed by a sum, each input row is multiplied with the corresponding weight and is then added to the sum computed by the previous input row.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{pictures/layer_outer_pipelining_ii.png}
    \caption{Partial schedule of the pipelined outer loop.}
    \label{fig:layer_outer_pipelining_ii}
\end{figure}

This addition takes 4 cycles and it must be applied for each input, so it is repeated \textit{IN\_SIZE}\footnote{equal to 256 too} times. This gives an approximate latency cycle count of $256 + 4 \times 256 = 1280$.

\textit{A quick note here: knowing the output is simply the sum of the effect of each input and the ordering of the sum has no effect, the latency could be reduced by summing the input contributions in a "tree-like" manner instead of a "chain-like" summation. The HLS tool is probably capable of such things but it goes outside of the scope of this course.}

\subsection{Area Oriented}
For area optimization, the sequential code of Sec. \ref{sec:stream_model} has been used to take advantage of the low resource consumption of FIFOs. To avoid duplicating computing elements, loop unrolling and pipelining have been disabled, together with the partitioning as parallel access is no longer needed. \\
The use of FIFOs removes the need for some flip flops (FF) where data does not need to be accessed multiple times, which makes it use less area.

\subsection{Results}
The 4 versions, differing only in the synthesis directives for the first 3 and with a slight code modification for the last one, were synthesized and the result is shown on Fig. \ref{fig:layer_results}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{pictures/layer_results.png}
    \caption{Result of the different versions for a single layer.}
    \label{fig:layer_results}
\end{figure}

The interval and latency for the inner and outer pipelining have already been explained and the last design has, without any surprise, worse performance metrics as they were traded for less components usage. The gain is mostly visible in the FF count. \\
Concerning the resources usage of the inner pipelining design, it only gains a few FFs and LUTs because only a small part of the code is pipelined. The outer pipelined design sees a major increase as the inner loop is unrolled, needing 256 times more resources\footnote{$256 \times 642 = 164.352$ being quite close to the true FF count.}. \\
Table \ref{tab:layer_performance_improvement} gives the improvement of each design with regards to the baseline and clearly show how much the outer pipelined design improved performance metrics and the FIFO design improved the area metrics.
